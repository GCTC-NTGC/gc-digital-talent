02 — Deployment

Post-deployment lifecycle

infrastructure/bin/post_deployment.sh runs as the startup command on every deployment. Steps are sequential and idempotent — re-running on an already-deployed slot is safe. Each step's outcome is reported to Slack.

01

Remove stale nginx apt source

Deletes /etc/apt/sources.list.d/nginx.list — a bad file shipped in the base image that breaks apt-get update. Can be removed from the script once the base image is updated.

02

Configure PHP CLI memory limit

Writes memory_limit=256M to a drop-in config file using overwrite (not append), so re-runs don't accumulate duplicate directives.

03

Install runtime packages

Installs supervisor, cron, and postgresql-client via apt. These are absent from the base image and required for queue workers, the scheduler, and DB connectivity.

04

Laravel optimize

Creates storage directories, sets permissions, prints the Lighthouse GraphQL schema, then runs php artisan optimize to compile the config, route, and view caches.

Config cache bakes env vars at this moment

If APP_URL has a trailing slash in App Settings when optimize runs, it appears in the cache with a trailing slash — causing double-slash redirects throughout the app. Fix the App Setting first, then run php artisan config:clear && php artisan optimize in the slot's SSH session.

05

Database migrations

Runs php artisan migrate --force --no-interaction against the shared PostgreSQL database. Output is captured and forwarded to Slack (capped at 2 500 characters). Because both slots share the DB, migrations run here affect live traffic immediately.

06

RolePermission seeder

Syncs role and permission records to match the current codebase definition. Safe to re-run — upserts, does not delete user role assignments.

07

Laravel Scheduler cron

Writes (overwrites) /etc/cron.d/gc-digital-talent — runs php artisan schedule:run every minute as www-data. Overwrite-on-run keeps it idempotent.

08

Supervisor setup

Runs setup_supervisor.sh — registers Laravel Horizon (queue workers) and Reverb (WebSockets) under supervisord, then starts the supervisor daemon.

09

nginx config reload

Substitutes $NGINX_PORT, $ROBOTS_FILENAME, and $HTTP_DISGUISED_HOST into the nginx config via substitute_file.sh, then reloads nginx. The CSP nonce placeholder (**CSP_NONCE**) — injected by Vite at build time — is handled separately by an nginx sub_filter at request-serve time.

# Verify nginx is up in the slot's SSH session $ curl -sk https://localhost:8080/ | head -3
10

Frontend env substitution

Runs substitute_file.sh on apps/web/dist/client/index.html with no variable list — replacing all environment variable placeholders with their live values. This is how APP_URL, OAUTH_CLIENT_ID, and other client-visible config reach the SPA without a frontend rebuild per environment.

03 — Identity

Canada Login OAuth requirements

Both slots connect to the same database. A user who logs in through the pilot slot and again through the main slot must resolve to the same database record — same roles, same profile, same history. This makes sub consistency a hard architectural requirement, not just a configuration detail.

Architectural requirement — consistent sub across clients

Because both slots share one database, the user record is the join point between them. Canada Login must return the same sub identifier for a given person regardless of which OAuth client they authenticated through. If Canada Login is configured for pairwise subjects (different sub per client), a user logging in through the pilot slot creates a separate database row — different from their main-slot record — with no role assignments. They appear unauthorized even after a successful login. Request public subject type (shared sub) explicitly when registering any new OAuth client for a slot.

RequirementFailure mode if missing
JWT access tokens isTokenProbablyExpired.ts calls jwtDecode() on every access token. Opaque tokens are not valid JWTs — the decode throws, the frontend treats the token as expired, no Authorization header is sent, and the backend returns null for every authenticated query. Request JWT tokens explicitly when registering the client with the CL team.
Matching OAUTH_CLIENT_ID CanadaLoginBearerTokenService::validateAndGetClaims() validates the JWT aud claim against the OAUTH_CLIENT_ID App Setting. If the slot's setting holds the wrong client ID, every JWT is rejected with a validation failure — the user is authenticated at CL but unauthorized in the app.
Shared sub across slots The user record in the shared database is keyed on sub. Role assignments, profile data, and application history all attach to that record. Because both slots read from the same database, a user logging in through the pilot slot must produce the same sub as they would through the main slot — otherwise they land on a blank record with no roles and appear unauthorized. This is the consequence of the shared-DB design and is non-negotiable.
Registered redirect URIs Both /en/auth/callback and /fr/auth/callback on the slot's domain must be registered with CL. A mismatch causes the OAuth flow to fail at the CL side before the app is involved.
03b — Shared Infrastructure

Analysing slot isolation before you replicate

Slots share a database but each has its own local filesystem. This combination produced a silent production failure with file downloads. The fix — DEPLOYMENT_SLOT_NAME as a unique queue name — is the pattern to apply whenever slots share a queue backend but not a filesystem. Before replicating this architecture, run the same analysis for every shared resource.

Lesson learned — filesystem locality + shared queue

Both slots share the same queue backend. Workers on either slot race to pick up every dispatched job. The problem: each slot has its own local filesystem. So what happened with large file downloads:

  1. Job inserted into the DB queue — dispatched from the main slot
  2. A free worker on the pilot slot raced and won the job
  3. The file was written to the pilot slot's local filesystem
  4. The success notification was stored in the shared database
  5. The main slot (serving 100% of traffic) read the notification from the DB and showed the download link — but the file existed on a different machine's disk
  6. User clicks download → 404. Logs also missing, because they were written to the other slot.

Fix: DEPLOYMENT_SLOT_NAME set to a unique value per slot, used as the Laravel queue name. Workers only consume from their own named queue — jobs dispatched on a slot are always processed by that slot's own workers, and the generated file lands on the same filesystem the user is served from.

Known follow-up — queue priorities for long-running jobs

File generation jobs share the same queue as short-lived jobs (emails, notifications, subscriptions). Because the file generation timeout was bumped to handle large datasets, retry_after must be set higher than that timeout — which means all jobs on the same queue, including fast ones, sit in the retry window longer than necessary if a worker crashes mid-file.

The right fix is a separate named queue for long-running jobs with its own higher timeout, while short-lived jobs stay on a standard queue with a short retry window. Workers can be given queue priority order so they drain urgent work first:

# Workers check 'high' queue first, then fall through to 'default', then 'low'
$ php artisan queue:work --queue=high,default,low

Both queue names should still incorporate DEPLOYMENT_SLOT_NAME as a prefix to maintain slot isolation — e.g. pilot-high, pilot-default.

Shared-infrastructure checklist for any slot architecture

For each resource both slots share, ask: can a request or job from one slot be processed or served by the other, and would that be incorrect? If yes, you need slot-level isolation for that resource.

Shared resourceRisk if not isolatedIsolation strategy
Queue backend (Redis / DB) Workers on either slot race to pick up every job. Each slot has its own local filesystem. A job that writes a file on slot B is invisible to slot A — the user sees a download link (from the shared DB notification) but gets a 404 (file on the wrong disk). We hit this. DEPLOYMENT_SLOT_NAME as the Laravel queue name, unique per slot. Workers only consume their own slot's queue.
Database A user logging in through the pilot slot creates a separate record if their sub differs — mismatched identity, missing roles. We hit this too. Canada Login public subject type — same sub across all clients. No DB-level change needed once CL is configured correctly.
Redis cache One slot caches a response that the other slot serves — stale or wrong-version data returned to users. Cache key prefix per slot: CACHE_PREFIX={DEPLOYMENT_SLOT_NAME} in Laravel config.
Blob / file storage Generated files (exports, uploads) written by one slot are readable by the other — version conflicts, overwritten outputs. Storage path prefix or container per slot using DEPLOYMENT_SLOT_NAME.
Session store A session created on the pilot slot is resumed on the main slot — auth state mismatch, unexpected logouts. If sessions are in Redis, apply the same key-prefix strategy as cache. Or use database sessions (already isolated by the shared-DB design).
04 — Replication

Deploying this pattern for another organization

A step-by-step guide for replicating this phased migration model in another government organization's production Azure App Service. The pilot slot approach lets you validate Canada Login with a small internal audience before any public-facing traffic is affected.

1

Create the App Service slot

In Azure Portal: App Service → Deployment slots → Add slot. Name it cl-pilot. Select Do not clone settings — you will configure them independently to avoid inheriting the wrong OAuth values.

# Or via Azure CLI $ az webapp deployment slot create \ --name <app-service-name> \ --resource-group <rg-name> \ --slot cl-pilot
2

Configure slot-specific App Settings

Portal → cl-pilot → Configuration → Application settings. Add these four and tick Deployment slot setting on each:

APP_URL = https://<pilot-custom-domain> OAUTH_CLIENT_ID = <cl-pilot-client-id> OAUTH_CLIENT_SECRET = <cl-pilot-client-secret> OAUTH_REDIRECT_URI = https://<pilot-custom-domain>/auth/callback
Tip

All other settings — database URL, mail config, CL discovery URI — should not be marked slot-specific. They inherit from the parent App Service and are shared across both slots.

3

Set the startup command

Portal → cl-pilot → Configuration → General settings → Startup Command:

post_deployment.sh $(SLACK_WEBHOOK_URI)

The script is included in the deployment artifact at infrastructure/bin/post_deployment.sh. Azure passes this to the container init process on every deployment and restart.

4

Register a Canada Login client for the pilot slot

Both slots share one database, so a user must resolve to the same record regardless of which slot they log in through. The CL client registration must reflect this. Contact the Canada Login team with these explicit requirements:

  • Subject type: public — shared sub across all clients for your organization. Not pairwise. This is mandatory for the shared-DB architecture to work correctly.
  • Access token format: JWT (not opaque / reference tokens)
  • Redirect URIs: https://<pilot-domain>/en/auth/callback and /fr/auth/callback
  • Scopes: same as the organization's main production CL client
5

Add the custom domain and TLS

Add the pilot domain as a custom hostname on the cl-pilot slot. Add a CNAME from your pilot subdomain to the slot's default Azure hostname (<app>-cl-pilot.azurewebsites.net). Bind TLS — App Service Managed Certificate works for non-production slots.

6

Deploy the artifact to the pilot slot

In your Azure DevOps pipeline, add a deployment step targeting the slot by name. The build artifact is the same zip produced by the existing pipeline — no separate build needed.

- task: AzureWebApp@1 displayName: 'Deploy to CL pilot slot' inputs: azureSubscription: '$(AZURE_SUBSCRIPTION)' appType: webAppLinux appName: '<your-app-service-name>' deployToSlotOrASE: true resourceGroupName: '$(RESOURCE_GROUP)' slotName: 'cl-pilot' package: '$(Pipeline.Workspace)/gcDigitalTalent/*.zip' startUpCommand: 'post_deployment.sh $(SLACK_WEBHOOK_URI)'
7

Verify the deployment

SSH into the pilot slot and confirm post-deployment completed cleanly:

# Confirm APP_URL has no trailing slash in config cache $ cd /home/site/wwwroot/api && php artisan config:show app | grep url # Confirm nginx is serving $ curl -sk https://localhost:8080/ | head -3 # Confirm the OAuth client ID in the config cache $ php artisan config:show services | grep oauth
Common issue — double-slash redirect

If login redirects to //en/applicant, the config cache has a trailing slash in APP_URL. Fix the App Setting, then SSH in and run: php artisan config:clear && php artisan optimize

05 — Deployment Model

Phased migration — not blue-green, not a slot swap

This architecture is a phased migration model. Each slot is an independent deployment target with its own CI/CD pipeline step. There is no slot swap. Users are not randomly split — they are deliberately onboarded group by group through a separate domain until the full migration is validated.

How the phased rollout worked

The goal was to migrate users to Canada Login gradually, with each group validating the integration before the next group onboarded. Issues surfaced and were resolved at each phase — the general public never experienced the bugs that earlier groups uncovered.

1

Pilot slot deployed and configured

The cl-pilot slot was created on the existing App Service. A separate Canada Login OAuth client was registered for the pilot domain. The same application code was deployed to both slots — the only difference was the slot-specific configuration (OAuth client, APP_URL, DEPLOYMENT_SLOT_NAME).

2

Phase 1 — Platform admins

Internal platform administrators were directed to the pilot domain. They authenticated through Canada Login on the pilot slot while general public traffic continued unaffected on the main slot. Issues found at this phase: opaque tokens (resolved by CL team), sub mismatch (CL configured pairwise instead of public — resolved), double-slash redirect (trailing slash in APP_URL — resolved in config).

3

Phase 2 — Power users

After the admin phase was stable, power users were onboarded to the pilot slot. This broadened the test surface — more varied workflows, more edge cases — while still keeping the general public on the unaffected main slot. The filesystem/queue race condition surfaced and was resolved at this phase (PR #16924).

4

Phase 3 — General public (main slot)

With all known issues resolved, the main slot was migrated to Canada Login. The general public onboarded to CL through the main domain. The migration was smooth — every failure mode had already been caught and fixed in the pilot phase. The pilot slot remained available as a staging environment for future CL-related testing.

Why this is not blue-green deployment

This architectureTrue blue-green
Database Both slots share one PostgreSQL database. A schema migration deployed to either slot affects both immediately. There is no isolation at the data layer. Each environment has its own independent database. The green environment can run a different schema version from blue without affecting live data.
Code versions Both slots run the same codebase. Deploying a new version to the pilot slot makes that version query the same shared database as the main slot — incompatible schema changes would break the main slot immediately. Blue and green can run entirely different application versions, including schema-incompatible changes. Rollback reverts both the code and the data layer.
Traffic routing Users are deliberately directed to a slot by domain — platform admins to the pilot domain, public to the main domain. There is no automatic traffic split or atomic cut-over. Traffic is atomically switched from blue to green. All users hit the new environment at once. Rollback is a second atomic switch.
Rollback Rolling back to the previous configuration requires a redeployment. Database migrations applied during deployment are not reverted — they must be designed to be backward-compatible. Rollback is instant — switch traffic back to blue. The blue database is still in the pre-migration state. No redeployment required.
Why this model was the right choice for CL migration

The goal was not to run two different versions of the application simultaneously — it was to validate Canada Login integration with a real but controlled user group before exposing the general public to it. A separate database per slot would have complicated user identity (the sub consistency requirement) without adding any benefit. The shared-DB model is the correct choice here: one database means one source of truth, and a user who logs in through either slot is always the same person with the same roles and history. The slot exists to give a controlled group of users a separately-configured OAuth entry point — not to run a different codebase.

Schema migration constraint

Because both slots share the database, any migration deployed to either slot takes effect immediately for both. Migrations must be designed to be backward-compatible with the code currently running on the other slot — the expand-contract pattern: add the new column before removing the old one, never drop a column in the same deployment that stops writing to it.
